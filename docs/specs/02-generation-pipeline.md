# 生成パイプライン仕様

ユーザー入力 → **ロジックによる**マークダウン形式の仮ページ（仕様書）の生成 → **仮ページを読んだ LLM（1 回のみ）**による実装 → **Git リポジトリ作成** までを定義する。デプロイは [Vercel デプロイ仕様](./03-vercel-deploy.md) に従う。

## 入力・出力

- **入力**: [入力スキーマ](./01-input-schema.md) に準拠した `SiteGenerationInput`（ページ数 1〜5・サイト説明・各ページのセクション・必須文言）。オプションで **添付画像**（キー `section_<page>_<block>` 形式）を渡せる。
- **中間成果物**: [1 本のマークダウン](./04-markdown-spec-format.md)（ページ構成の仕様＋仮ページを 1 ファイルにまとめたもの）
- **最終成果物**: ビルド可能な Next.js プロジェクトが入った 1 つのディレクトリ。かつ Git リポジトリとして初期化済み（必要に応じてリモート作成済み）。添付画像は `public/images/` に `<ref>.<ext>` で配置され、生成コードから `/images/<ref>.<ext>` で参照される。

## パイプラインのステップ

### フェーズ 1: 1 本のマークダウン（仕様＋仮ページ）の生成（ロジック）

1. **入力の検証**  
   `SiteGenerationInput` をバリデーションし、不正ならエラーを返して終了。

2. **ロジックによる 1 本マークダウン生成**  
   **LLM は使わない**。ユーザー入力を [04-markdown-spec-format.md](./04-markdown-spec-format.md) に従い、**ページ構成の仕様** と **仮ページ** を **1 本のマークダウン** に組み立てる。  
   - おまかせ時の固定文言は [04 の正式定義](./04-markdown-spec-format.md#固定文言の正式定義) に従う。  
   - 同一入力なら **同一の 1 本マークダウン** が得られる（再現可能）。

3. **1 本マークダウンの保存・返却**  
   生成されたマークダウンをファイルとして保存するか、メモリ上で保持する。  
   - ユーザーへの表示・ダウンロード用と、後段の「実装」ステップへの入力用の両方に使う。

### フェーズ 2: 仮ページに基づく実装（LLM・1 回のみ）

4. **LLM による実装**  
   1 本のマークダウン（仕様＋仮ページ）を **LLM に 1 回だけ**入力として渡し、[07 生成サイト技術仕様](./07-generated-site-tech-spec.md)（Next.js / Tailwind / Vercel 固定）に従ってソースコードを生成させる。利用する LLM は [08 LLM 選定](./08-llm-selection.md) で実装前に決める。  
   - 出力はファイル群（`package.json`, `app/`, `components/`, スタイル等）。TypeScript・npm を前提とする。  
   - 仕様書に書かれた見出し・本文・文言・画像指定をすべて反映する。  
   - **添付画像**: 呼び出し元が渡した添付画像は、LLM の出力後に **`outDir/public/images/`** に `<ref>.<ext>`（例: `section_0_0.png`）で書き出される。LLM には「添付画像は public/images/ に配置するので、`next/image` で `/images/<ref>.<ext>` を参照すること」と指示する。  
   - **LLM 最適化**: コンテキストは **1 本のマークダウンと技術仕様（07）の要約のみ**。既存のデザインコードは渡さない。詳細は [06-llm-optimization.md](./06-llm-optimization.md)。

5. **ビルドの可否確認**  
   生成ディレクトリで `next build`（または `pnpm build`）を実行し、成功することを確認する。失敗した場合はエラーを返す（オプション: リトライや LLM へのフィードバックは別仕様で検討）。

6. **出力ディレクトリの返却**  
   生成された Next.js プロジェクトのディレクトリパスを、呼び出し元に返す。

### フェーズ 3: Git リポジトリの作成

7. **Git 初期化**  
   生成ディレクトリで `git init` を実行し、全ファイルをコミットする。

8. **リモートの扱い（オプション）**  
   Vercel が Git 連携でデプロイする場合は、リモートリポジトリ（GitHub/GitLab/Bitbucket 等）を作成し、push する。  
   ZIP アップロードでデプロイする場合は、ローカル Git のみで完了とする。  
   どちらを採用するかは [03-vercel-deploy.md](./03-vercel-deploy.md) および実装方針に合わせる。

## コンポーネントの責務

- **1 本マークダウン生成**: `packages/core` が **ロジック**（テンプレート＋入力の組み立て）で担当。LLM は呼ばない。
- **実装生成**: `packages/core` または API が、1 本のマークダウンを **LLM に 1 回だけ**渡し、[07](./07-generated-site-tech-spec.md) に従ったコードをファイルとして書き出す。
- **Git 操作**: `packages/deploy` または専用パッケージが、生成ディレクトリに対して `git init` と `git add` / `git commit` を実行。必要ならリモート作成・push。

## 冪等性・再現性

- **1 本のマークダウン**はロジックで生成するため、同じ `SiteGenerationInput` から **常に同一** が得られる。
- **実装**は LLM の非決定的な振る舞いの影響を受けるため、完全な再現は保証しない。必要に応じてシードや温度パラメータで制御する。

## パイプラインの実行主体

Web UI が呼び出すのは **単一のエントリポイント** とする（例: `packages/core` の `runFullPipeline()` または `apps/web` の API ルート／`apps/api` の 1 エンドポイント）。このエントリポイントが「入力検証 → 1 本 MD 生成 → LLM 実装 → ビルド確認 → Git 初期化 → [03 に従い] Vercel デプロイ」までを一括で実行し、成功時のみ URL・失敗時はエラーを返す。成功／エラーの判定責任はこのエントリポイントが持つ。

- **実行形態**: 初版は **同期** とする。実行環境がサーバーレス（Vercel 関数等）でタイムアウト（例: 60 秒）に収まらない場合は、非同期ジョブ＋ポーリングを [02-system-architecture](../architecture/02-system-architecture.md) に従い検討する。
- **タイムアウト**: 成功基準「10 分以内に URL 取得」に合わせ、パイプライン全体の最大待機時間を **10 分** とし、超過時はエラーを返して [09 エラーページ](./09-generator-ui-spec.md) に遷移させる。

## 呼び出し元（Web UI）への返却契約

バックエンド（上記エントリポイント）は、**デプロイ完了かつ URL が取得できた場合にのみ**「成功・URL」を返す。途中で失敗した場合は **成功レスポンスを返さず、エラーを返す**。これにより [09 ジェネレーター UI](./09-generator-ui-spec.md) の「完了ページはエラーが無い場合のみ表示」が保証される。

- **成功時**: `{ url: string; specMarkdown?: string }`。`specMarkdown` は生成した 1 本マークダウン（文字列）で、[09 完了ページ](./09-generator-ui-spec.md) で仕様書のダウンロード用に利用する。省略可。
- **エラー時**: `{ error: string; code?: string }`。`error` はユーザーに表示するメッセージ、`code` は種別（例: `VALIDATION_ERROR`, `LLM_ERROR`, `DEPLOY_ERROR`）でエラーページでの表示やリトライ判定に使う。

## LLM の性質に最適化した設計

仮ページ生成に LLM を使わず、実装フェーズで **1 回だけ** LLM を呼ぶ設計は [06-llm-optimization.md](./06-llm-optimization.md) に定義する。実装時は必ず参照すること。
