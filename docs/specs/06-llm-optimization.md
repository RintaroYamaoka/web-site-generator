# LLM の性質に最適化した設計

本プロジェクトでは **仮ページ（仕様書）はロジックで生成し、LLM は実装の 1 回だけ** 使う。**実装を始める前に [08 LLM 選定](./08-llm-selection.md) で利用するプロバイダ・モデルを決めておく。** 実装時は以下を守る。

## 前提：なぜ仮ページをロジックで作るか

- **設計図を LLM が処理したものをさらに LLM に読み込ませると、ブレが二重になる**。1 台目の出力が毎回少しずつ変わり、2 台目がその「毎回違う設計図」を解釈するため、全体のばらつきが大きくなる。
- 仮ページを **ロジック**（テンプレート＋入力の組み立て）で作れば、同一入力で常に同じ仮ページになる。LLM は「その 1 通りに決まった設計図 → コード」の **1 回だけ** なので、ブレの原因は実装フェーズに限定され、再現性・コスト・意図の保持がしやすい。

## LLM を使うのは「実装」の 1 回のみ

- **仮ページ生成**: LLM は使わない。ロジックで [04 のフォーマット](./04-markdown-spec-format.md) に沿ったマークダウンを組み立てる。
- **実装**: 仮ページ（マークダウン）を LLM に入力として渡し、Next.js プロジェクトを生成する。**ここだけ** LLM API を呼ぶ。

## 実装フェーズでのコンテキスト

- **渡すもの**: **1 本のマークダウン**（ページ構成仕様＋仮ページ）と、[07 生成サイト技術仕様](./07-generated-site-tech-spec.md)の要約（Next.js / Tailwind / Vercel / TypeScript / npm 固定）、および「必須ファイル構成・出力形式」のテンプレート（`package.json`, `tsconfig.json`, `next.config`, `app/layout.tsx`, `app/page.tsx`, `app/form/page.tsx` など）。
- **渡さないもの**: 既存のサイトのコード、デザインシステムのコード、長いテンプレート。参照するなら「ディレクトリ構造・**固定したいファイル名リスト**」程度に留め、**見た目やスタイルのコードは渡さない。**
- **理由**: 実装 LLM のコンテキストは「この中身をどう見せるか」＋「守るべき技術仕様・ファイル構成」に絞る。ファイル構成と技術仕様をテンプレートで固定することで、**import ミスや不足ファイルによるビルドエラーを減らす**。

## 役割と出力形式・ファイル構成の固定

- **実装フェーズ**: 役割は「実装者」。入力は「1 本のマークダウン（仕様＋仮ページ）」。出力は **ファイル群（コード）**。[07 生成サイト技術仕様](./07-generated-site-tech-spec.md)に従い、Next.js（App Router）・Tailwind・TypeScript・npm で実装する。プロンプトで「仕様書の見出し・本文・文言・画像指定をすべて反映すること。デザインは中身とデザイン希望に合わせる」と明示する。
  - 出力形式は `---FILE: 相対パス--- ... ---END---` に統一し、**必須ファイル（package.json / tsconfig.json / next.config / tailwind.config / app 配下の主要ページなど）をすべて出力することを明示**する。
  - 「新しいルートや不要なページを勝手に増やさない」「仕様書にないパスは極力作らない」ことをプロンプトで指定する。

## 実装時のチェックリスト

- [ ] 仮ページ生成に LLM を呼んでいないか（ロジックのみ）
- [ ] 実装のプロンプトに、1 本のマークダウンと [07](./07-generated-site-tech-spec.md) 以外の「完成サイトのコード」をコンテキストとして渡していないか
- [ ] 実装フェーズで「仕様書の内容をすべて反映し、技術は Next.js / Tailwind / TypeScript に固定する」と指示しているか
