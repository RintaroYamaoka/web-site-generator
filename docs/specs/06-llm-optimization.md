# LLM の性質に最適化した設計

本プロジェクトのパイプライン（仮ページ生成 → 実装）は、**LLM がコンテキストに強く引きずられる**という性質を前提に設計する。実装時は以下を守る。

## 前提：LLM の性質

- **与えられたテキスト・コードをコンテキストとして強く参照する**。特にコードを渡すと、その構造・スタイル・技術選定に沿った出力をしがち。
- **役割と出力形式をはっきりさせたほうがブレにくい**。「何を入力とし、何だけを出力するか」を厳密にすると、中身とデザインの噛み合わせが良くなる。

## 原則 1：フェーズごとにコンテキストを分離する

### フェーズ 1（仮ページ生成）

- **渡すもの**: ユーザー入力（ページ数・サイト説明・セクション・必須文言）と、[仮ページのフォーマット](./04-markdown-spec-format.md)の説明のみ。
- **渡さないもの**: 既存の Next.js コード、デザインサンプル、テンプレートのコード、CSS/コンポーネント例。**デザインや実装のコードは一切コンテキストに含めない。**
- **理由**: デザインを入れると LLM がそれに引きずられ、中身（文言・構成）が二の次になりやすい。**中身だけ**を生成させる。

### フェーズ 2（実装）

- **渡すもの**: **仮ページ（マークダウン）** と、技術要件（Next.js App Router、アクセシビリティ、モバイル対応など）の短文。必要なら「トーン: フォーマル」など仮ページ先頭のメモ。
- **渡さないもの**: 既存のサイトのコード、デザインシステムのコード、長いテンプレート。参照するなら「ディレクトリ構造・ファイル名の例」程度に留め、**見た目やスタイルのコードは渡さない。**
- **理由**: 実装 LLM のコンテキストは「この中身をどう見せるか」だけにすべき。既存デザインコードを渡すと、仮ページの内容よりそのコードに合わせた実装になりやすい。

## 原則 2：役割と出力形式を固定する

- **フェーズ 1**: 役割は「コンテンツ設計者」。出力は **マークダウンのみ**（コードブロックで囲むならマークダウン本文だけ）。HTML や React は出さない。
- **フェーズ 2**: 役割は「実装者」。入力は「仮ページ（マークダウン）」。出力は **ファイル群（コード）**。プロンプトで「仮ページに書かれた見出し・本文・文言をすべて反映すること。デザインは中身に合わせて決める」と明示する。

## 原則 3：2 段階を 1 回の呼び出しにまとめない

- 仮ページ生成と実装を **別々の LLM 呼び出し** にする。1 回で「マークダウンもコードも」出させない。
- 理由: 1 回で両方出すと、モデルが先にコードを書いてからマークダウンを合わせる、あるいはコードに引きずられた中身になりやすい。**中身 → デザイン** の順をコンテキスト上も保つ。

## 実装時のチェックリスト

- [ ] 仮ページ生成のプロンプトに、Next.js や CSS のコード例を含めていないか
- [ ] 実装のプロンプトに、仮ページ（マークダウン）以外の「完成サイトのコード」をコンテキストとして渡していないか
- [ ] 実装フェーズで「仮ページの内容をすべて反映し、デザインは中身に合わせる」と指示しているか
- [ ] フェーズ 1 とフェーズ 2 で、別リクエスト・別コンテキストとして呼び出しているか
