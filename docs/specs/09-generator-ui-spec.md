# ジェネレーター UI 仕様（apps/web）

本プロジェクトの **ツール側**（`apps/web`）の画面構成・世界観・UX を定義する。入力スキーマは [01-input-schema.md](./01-input-schema.md) に従う。

## 世界観・デザイン方針

- **洗練された無機質** … ニアオートマタ（NieR: Automata）をイメージした、機械的でクリーンな世界観。
- **シンプルで凝ったデザイン** … 装飾を削ぎ落としつつ、アニメーションやタイポグラフィ・余白で「質」を出す。
- **誰でも直感的にサイトが作れる** … 専門用語を避け、操作の流れが一目で分かる UI/UX。

---

## 画面構成と遷移

```
[トップ] ──「Webサイトを作成」──→ [入力フォーム] ──「サイト作成」──→ [ローディング]
                                                                           │
                    成功（URL 発行・エラーなし）←─────────────────────────┤
                    エラーあり ──────────────────────────────────────────→ [エラーページ]
                     │
                     └──────────────────────────────────→ [完了ページ：URL 表示]
```

- **トップ** … シンプルな構成。最低限の説明と「Webサイトを作成」ボタン。
- **入力フォーム** … ページ数・各ページのセクション・必須文言・デザイン希望を入力。ページ数などに応じて動的にフォームが変わる（案は後述）。
- **ローディング** … サイト生成中。「お待ちください」＋ AI のニューロンネットをイメージした無機質なアニメーション。
- **完了ページ** … **デプロイ完了かつエラーが無い場合のみ**表示。発行された URL をシンプルかつ凝ったデザインで表示。
- **エラー時** … **エラー専用ページ**に遷移する。バックエンド／フロントの状態管理で「成功」にしないため、エラー時は完了ページ（URL 表示）には遷移しない。

### バックエンド側の責務

- デプロイ完了・URL 取得・途中の失敗（LLM 失敗・Git 失敗・Vercel 失敗など）を明確に区別し、**エラーが発生した場合は「成功＋URL」を返さない**。
- フロントは「成功レスポンスかつ URL あり」のときだけ完了ページに遷移する。

## 各画面の仕様

### 1. トップページ

- **構成** … シンプル。ロゴまたはタイトル、ごく短い説明文（1〜2 文）、「Webサイトを作成」の CTA ボタン。
- **デザイン** … 無機質で洗練された世界観（ニアオートマタ風）。余白を効かせ、色数は抑える（モノトーン＋アクセント 1 色程度）。
- **説明文** … 「デモ用サイトを数分で生成」「ページ数と内容を入力するだけ」など、最低限の説明に留める。

### 2. 入力フォームページ

- **遷移** … トップの「Webサイトを作成」クリックでこのページへ。
- **目的** … [入力スキーマ](./01-input-schema.md) に沿った内容を、**誰でも直感的に**入力できるようにする。
- **動的 UI** … ページ数・セクション数などに応じてフォームが動的に変わる。案は下記「動的フォームの案」を参照。
- **CTA** … 「サイト作成」ボタンで送信。押下後は**即座にローディングページへ遷移**し、完了時に結果 or エラーページへ `replace` で遷移する（戻るでローディングに戻らない）。
- **フォーマット・テキスト色の統一** … フォームの視認性と統一感のため、次のルールで揃える。
  - **項目名・見出し**（サイトの概要、ページ 1、サイトのページ数 等）: `--fg`、`text-sm font-medium`。
  - **補足・説明・例**（「例: トップ・会社概要…」「ヒーローセクション。不要な場合は…」等）: `--muted`、`text-xs`。
  - **入力欄の値（ユーザーが入力した文字）**: `--input-fg`（#ffffff の白文字）。プレースホルダーと区別し、仮ページに送る実入力であることを明確にする。
  - **プレースホルダー**: `--muted`（CSS の `::placeholder` で統一）。例文は**placeholder のみ**に表示し、初期値は空欄とする（入力欄に例文を初期値として持たない）。これにより仮ページに「例: …」がそのまま出ることを防ぐ。
- **ページ別プレースホルダー（例文）** … ページ数を増やしたとき、各ページの 1 ブロック目（ヒーロー）のページ名・見出し・本文の**placeholder**に次の例文を表示する。入力欄の初期値は空。画像はデフォルトで「AIおまかせ」。
  - 1 ページ目: 名前「例: トップ」、見出し「例: 株式会社XX」、本文「例: キャッチコピー。空欄でAI補完・おまかせ可。」
  - 2 ページ目: 「例: 会社概要」／「例: 会社概要」／「例: 会社概要の説明文。空欄でAI補完・おまかせ可。」
  - 3〜5 ページ目: 同様に「例: 実績」「例: サービス紹介」「例: お問い合わせ」等。
  - 6 ページ目以降（将来拡張時）: 名前「例: ページ名」、見出し「例: セクション見出し」、本文「例: このページの概要テキスト。空欄でAI補完・おまかせ可。」
- **入力内容をクリア** … 「サイトを作成する」ボタンと横並びで「入力内容をクリア」ボタンを配置する。押下時に「入力内容をすべてクリアします。よろしいですか？」と警告表示し、了解時のみ全項目を初期状態に戻し、localStorage の下書きも削除する。
- **ブロック追加** … 「+ ブロックを追加」は常に**そのページのセクション末尾**に追加する（ヒーロー有無にかかわらず下に追加）。
- **下書きの永続化** … リロードで入力が飛ばないよう、フォーム内容（ページ数・サイト概要・ヘッダー・フッター・各ページ・デザイン希望）を **localStorage** に下書き保存する。キーは `web-site-generator-form-draft`。画像添付（File）は保存できないため、リロード後は画像のみ再選択が必要。初回表示時は保存データの読み込みが終わるまで「読み込み中…」を表示してからフォームを描画する。
  - **実装上の制約（Next.js / SSR）** … 初回レンダーはサーバーでも行われるため `window` / `localStorage` は使えない。**復元はクライアントの `useEffect` でマウント後に 1 回だけ**行う。**復元が完了するまで保存処理は行わない**（デフォルト値で上書きしない）。復元完了フラグを立ててから保存を有効にし、保存は入力変更をデバウンス（例: 1 秒）して行う。
  - **動作順序** … ① マウント → ② クライアントで localStorage から読み取り → ③ 有効な下書きなら state に反映 → ④ 復元完了 → ⑤ 以降、入力変更をデバウンスして保存。「読み込み中…」は ④ まで表示する。
- **添付画像の実利用** … フォームでアップロードした画像は、送信時に `section_<page>_<block>` 形式のキーで FormData に含め、API が `runFullPipeline(input, { attachedImages })` に渡す。パイプラインは [02 生成パイプライン](./02-generation-pipeline.md) に従い、LLM 出力後に `public/images/` へ書き出し、生成サイトでは `/images/<ref>.<ext>` で参照される。

### 3. ローディングページ（生成中）

- **表示タイミング** … 「サイト作成」送信後、デプロイ完了 or エラーが返るまで。
- **内容** … 「サイトを生成しています。しばらくお待ちください」などの文言。
- **デザイン** … **AI のニューロンネットをイメージしたアニメーション**（ノードと接続が光る、パルスする等）。無機質で機械的な印象。
- **エラー時** … ローディングは終了し、**エラー専用ページ**に遷移する。

### 4. 完了ページ（URL 表示）

- **表示条件** … **デプロイが完了し、URL が発行され、かつエラーが無い場合のみ**表示。バックエンドが成功を返し、フロントがその結果だけを完了ページに使う。
- **内容** … 発行された URL を大きく・コピーしやすく表示。必要なら「開く」「コピー」ボタン。**仕様書（1 本マークダウン）のダウンロードリンク**を用意し、ユーザーが生成に使った仕様書を確認できるようにする（US-2 の「表示またはダウンロード」を満たす）。
- **デザイン** … シンプルで無機質だが凝ったデザイン。世界観はトップ・ローディングと統一。

### 5. エラーページ

- **表示条件** … 生成・デプロイのいずれかでエラーが発生した場合。バックエンドが [02 のエラー形式](./02-generation-pipeline.md#呼び出し元web-uiへの返却契約)（`error`, `code`）を返したとき、ローディングページからこのページへ遷移する。
- **内容** … エラー内容（`error` をメッセージとして表示。`code` があれば種別表示やリトライ可否の判断に利用）を表示。**リトライ**（入力フォームに戻って再送信）や**トップに戻る**の導線を用意する。
- **リトライ時の入力復元** … リトライで入力フォームに戻る際、セッションまたは localStorage に保存済みの直前入力があれば **復元して表示**する。ユーザーが編集し直しやすくする。
- **デザイン** … シンプルで無機質。世界観は他画面と統一しつつ、エラーであることが分かる表現（文言・アイコン等）にする。

## 動的フォームの案（入力フォームページ）

入力項目（ページ数 1〜5、各ページのセクション・内容・画像・必須文言・デザイン希望）を、直感的に編集できるようにするための案。

### 案 A: ページ数ファースト ＋ ページごとブロック

1. **最初に「ページ数」を選択**（1〜5）。ラジオまたはセレクト。
2. **選択した数だけ「ページ」ブロックを表示**。各ブロックには「パス」「ページ名」と「セクション」のリストを配置。
3. **各ページ内で「セクションを追加」**でセクションを動的に追加。各セクションには見出し・内容（テキスト or おまかせ）・画像（添付 or テキスト or おまかせ）を入力。
4. **共通項目**はフォームの前か後ろに一括で：必須文言（複数可）、デザイン希望（テキスト or おまかせ）。

- **メリット** … 今どこのページ・セクションを編集しているかが分かりやすい。ページ数を変えるとブロックが増減するので、入力スキーマと一致しやすい。
- **実装** … `pageCount` 変更で `pages` の長さを合わせ、足りない分はデフォルト（例: パス `/page2`、名前「ページ2」、セクション 1 件）で補う。

### 案 B: ステップウィザード形式

1. ステップ 1: ページ数・サイトの説明・プロジェクト名。
2. ステップ 2: 各ページのパス・名前・セクション数（または「セクション追加」で都度追加）。
3. ステップ 3: 各セクションの詳細（見出し・内容・画像）。ページごとにサブステップでもよい。
4. ステップ 4: 必須文言・デザイン希望 → 「サイト作成」。

- **メリット** … 1 画面に詰め込まないので、初見でも進めやすい。
- **デメリット** … ステップ数が多く、ページ・セクション数によっては「戻って直す」が発生しやすい。

### 案 C: 1 画面 ＋ アコーディオン／タブ

- 左（または上）に **ページ一覧**（ページ 1〜5 のタブ or アコーディオン）。ページ数を選ぶと表示されるページ数だけタブが有効になる。
- 右（または下）に **選択中ページのセクション一覧**。各セクションはアコーディオンで開閉。セクション追加ボタンで動的追加。
- 必須文言・デザイン希望は固定で下部またはサイドに常時表示。

- **メリット** … 全体を一覧しやすく、ページ切り替えが速い。
- **デメリット** … 画面が横に広い、または縦に長くなる。モバイルではタブ・アコーディオンの設計が重要。

### 推奨

- **案 A** をベースにすると、実装がシンプルで [01 入力スキーマ](./01-input-schema.md) との対応も取りやすい。ページ数変更でブロックが増減し、各ページ内でセクションを追加・削除する形にすると、直感的かつ動的になる。
- セクション数の上限は UI で規定する（例: 1 ページあたり 1〜10）。上限は [01-input-schema.md](./01-input-schema.md) の「別途 UI で定義可」に合わせて本仕様で決めてよい。

## セクション上限（UI 側）

- 1 ページあたり **セクション数は 1 以上・10 以下** とする（目安）。必要に応じて仕様で変更可能。
- バリデーションはフロントと [01 入力スキーマ](./01-input-schema.md) を実装する `packages/core` または `packages/shared` で共通化する。

## 実装との対応

- **テキスト色・CSS** … `apps/web/app/globals.css` で `--fg` / `--muted` を定義し、`input::placeholder` / `textarea::placeholder` を `--muted` に統一。フォーム各要素には必要に応じて `text-[var(--fg)]` を明示。ヒーロー欄の初期値がデフォルト例文と一致するときは `getDefaultHeroForPage(i)` で判定し muted を適用（全ページ共通）。
- **送信後の遷移** … `apps/web/app/form/page.tsx` の handleSubmit で送信直後に `router.push("/loading")`、成功時は `router.replace("/result?...)`、失敗時は `router.replace("/error?...)` とする。
- **ページ別プレースホルダー・初期値** … `PAGE_PLACEHOLDERS` と `getPlaceholderForPage(i)`（`form/page.tsx`）。初期値は空（`defaultPage` の name/heading/content は `""`）。ブロック追加は `addSection` で常に `[...page.sections, newSection]`。入力値の色は `--input-fg`（globals.css で #ffffff）。クリアは `handleClear` で confirm 後に state リセット＋localStorage 削除。
- **下書き保存** … `apps/web/app/form/formDraftStorage.ts` に `loadDraft(storage)` / `saveDraft(storage, draft)` / `isValidDraft(draft)` を切り出し、ストレージを差し替えてテスト可能にする。フォームはマウント後 1 回だけ `loadDraft(localStorage)` して通過時のみ state に反映し、復元完了フラグを立てる。保存は復元完了後のみ有効にし、入力変更を 1 秒デバウンスして `saveDraft` を呼ぶ。復元完了まで「読み込み中…」を表示。
