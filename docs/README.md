# ドキュメント駆動開発 — 本プロジェクトの進め方

## 方針

**コードを書く前にドキュメントで設計する**ことを原則とします。

- 要件・目標は `requirements/` に記載し、実装前に合意する
- アーキテクチャ・ディレクトリ構造は `architecture/` に定義する
- 詳細仕様は `specs/` に書き、実装のたびに参照・更新する
- 重要な判断は `decisions/` に ADR として残す

## ドキュメント構成

| ディレクトリ | 役割 | 更新タイミング |
|-------------|------|----------------|
| [requirements/](./requirements/) | 目標・ユーザーストーリー・受け入れ条件 | フェーズ開始時・要件変更時 |
| [architecture/](./architecture/) | システム構成・ディレクトリ設計・データフロー | 設計・リファクタ時 |
| [specs/](./specs/) | API・UI・生成パイプラインの詳細仕様 | 実装前・仕様変更時 |
| [decisions/](./decisions/) | アーキテクチャ決定記録（ADR） | 重要な技術判断時 |
| [templates/](./templates/) | サイトテンプレート・コンテンツの設計メモ | テンプレ追加・変更時 |

## 開発フロー

1. **要件** → `requirements/` を更新し、何を作るか確定する
2. **設計** → `architecture/` と `specs/` で「どう作るか」を書く
3. **実装** → ドキュメントを唯一の仕様書としてコードを書く
4. **変更** → 仕様が変わったらまずドキュメントを更新し、その後コードを合わせる

## ドキュメントの参照順（推奨）

新規参加者・実装開始時は次の順で読むと理解しやすいです。

1. [プロジェクト概要・目標](./requirements/00-overview.md)
2. [ディレクトリ構造](./architecture/01-directory-structure.md)
3. [システムアーキテクチャ](./architecture/02-system-architecture.md)（仮ページ生成 → LLM 実装 → Git → Vercel の流れ）
4. 該当機能の [specs/](./specs/) 内ドキュメント（入力スキーマ・生成パイプライン・[マークダウン形式の仮ページ](./specs/04-markdown-spec-format.md) 等）
