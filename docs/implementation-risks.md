# ドキュメント駆動で実装した場合に起こりうる問題（調査結果）

本ドキュメントは、現時点の `docs/` を「正」として実装を進めたときに発生しうる**齟齬・曖昧さ・未定義**を列挙する。実装前の解消または実装時の判断材料として利用する。

---

## 確認が必要なこと（ユーザー判断）

実装を進めるにあたり、**仕様で決めるべき項目はすべて反映済み**です。08 は無料枠でコード生成向けに推奨される **Google / Gemini 3 Flash Preview** で記載済み。変更したい場合は [08-llm-selection.md](./specs/08-llm-selection.md) を編集してください。

---

## 1. ドキュメント間の矛盾・不明確さ

### 1.1 templates/next-demo の役割

- **01-directory-structure.md**: `templates/next-demo` を「`packages/core` がここをコピーし、動的データで上書き・追加する」と定義している。
- **02-generation-pipeline.md / 07-generated-site-tech-spec.md**: LLM が **1 本のマークダウンから Next.js プロジェクトを一から生成**するフローで、**next-demo のコピーは登場しない**。
- **docs/templates/README.md**: `config/pages.json` を書き換える「設定駆動」のテンプレート利用を想定している。

**起こりうる問題**: 実装者が 01 に従って core で next-demo をコピーしつつ、02 に従って LLM にコード生成させる二重の流れを作ってしまう。あるいは next-demo を無視して LLM のみで実装し、01 の記述が陳腐化する。

**→ 対応済み**: [01-directory-structure.md](./architecture/01-directory-structure.md) で `next-demo` を「現行フローでは未使用。将来のテンプレートベース生成用」と明記し、用語を「マークダウン組み立て用テンプレート」と「Next.js 雛形」で区別した。

---

### 1.2 「テンプレート」の指すものの揺れ

- **06 / 02**: 「テンプレート＋入力の組み立て」＝ **1 本マークダウンを組み立てるロジック**（文字列テンプレートや構造のひな形）を指している。
- **01-directory-structure**: 「テンプレート」＝ **Next.js プロジェクトの雛形**（next-demo）を指している。

**起こりうる問題**: 「テンプレート」という語が文脈で異なる意味になり、会話や実装で混乱する。

**→ 対応済み**: 01 の core の説明で「マークダウン組み立て用テンプレート＋入力」と表記し、next-demo は「Next.js 雛形」として区別した。

---

## 2. 未定義・仕様不足

### 2.1 パイプラインの実行主体と UI への返却

- **02**: 「呼び出し元（Web UI）への返却契約」で、デプロイ完了かつ URL 取得時のみ成功を返すとある。
- **02 / architecture**: パイプラインのステップは「入力検証 → 1 本 MD 生成 → LLM 実装 → ビルド確認 → Git → (03 に従い) デプロイ」だが、**誰が一連の流れを実行し、誰が Web UI に成功/エラーを返すか**が明文化されていない（core 単体か、core + deploy を呼ぶ API か、apps/web が順次呼ぶか）。

**起こりうる問題**: 実装者が「core は MD とコード生成まで」「deploy は Git と Vercel だけ」と分離し、UI が両者を別々に呼んだ場合、**「成功はデプロイ完了時のみ」**を満たす責任が曖昧になる。タイムアウトや途中失敗時の扱いも揃いにくい。

**→ 対応済み**: [02-generation-pipeline.md](./specs/02-generation-pipeline.md) に「パイプラインの実行主体」を追加。単一エントリポイントが一括実行し、成功時 `{ url, specMarkdown? }`・失敗時 `{ error, code? }` を返す形で定義。10 分タイムアウト・初版は同期の注記も追加。

---

### 2.2 実行形態とタイムアウト

- **02-system-architecture**: 長時間ジョブの場合は `apps/api` で非同期ジョブ＋ポーリングを検討するとあるが、**初版で同期呼び出しにするか非同期にするか**が決まっていない。
- LLM 呼び出し＋ビルド＋Git＋Vercel デプロイは **数分** かかる可能性が高く、Vercel のサーバーレス関数は **10〜60 秒** でタイムアウトする。

**起こりうる問題**: apps/web の API ルートから同期的に core → deploy を呼ぶ実装にすると、本番でタイムアウトし、09 の「ローディング → 完了/エラー」が実現できない。

**→ 対応済み**: 02 で初版は同期、最大待機 10 分でタイムアウトしてエラー返却。サーバーレスで収まらない場合は非同期ジョブを検討する旨を記載。

---

### 2.3 画像添付のエンドツーエンド

- **01**: `image.type === "attach"` のとき `fileRef` を必須とし、「パイプラインが LLM に渡す際に実体を紐付ける」とある。
- **04**: 添付画像は「LLM 入力時に実体を渡す」とある。
- **06**: 渡すものは「1 本のマークダウン」と技術仕様の要約であり、**マルチモーダル（画像バイナリや base64）の渡し方**は書かれていない。

**起こりうる問題**: 実装時に「画像付きで LLM を呼ぶ」手順が仕様から一意に決まらない。API の multipart 受け取り、ref とファイルの対応、LLM API の画像入力形式（data URL / ファイルパス / 別パート）が揃わない。

**推奨**: 「画像添付時のデータフロー」を 01 または 02 に短く定義する（UI が multipart で送る、core/API が ref と実体を保持、LLM 呼び出し時にマークダウン＋画像を所定形式で渡す）。利用 LLM のマルチモーダル仕様（08 選定結果）に依存するため、08 確定後に追記するとよい。

---

### 2.4 固定文言の単一ソース

- **01 / 02 / 04**: おまかせ時の固定テキストとして「例: `AIにおまかせ`」「`画像: おまかせ`」が複数ファイルに「例」として書かれているが、**正式な値が 1 か所で定義されていない**。

**起こりうる問題**: core のロジックと LLM 用プロンプト（「この文言はおまかせを表す」）で表記が微妙にずれ、解釈のブレやテストの期待値が合わない。

**→ 対応済み**: [04-markdown-spec-format.md](./specs/04-markdown-spec-format.md) に「固定文言の正式定義」を追加。`AIにおまかせ` / `画像: おまかせ` を正式値として定義し、02 から参照するようにした。

---

### 2.5 エラー応答の形式

- **02**: 失敗時は「成功レスポンスを返さず、エラーを返す」。
- **03 / 05**: 「呼び出し元がメッセージをユーザーに表示できる形で返す」。
- **09**: エラーページで「エラー内容（メッセージや種別）」を表示する。

**起こりうる問題**: エラー時のレスポンス形（`{ error: string }` / `{ code, message, details }` / HTTP ステータスのみなど）が定義されておらず、フロントがメッセージや種別をどう取り出すか実装者任せになる。

**→ 対応済み**: 02 でエラー時は `{ error: string; code?: string }` と定義。09 エラーページでこの形式を参照して表示する旨を追記。

---

### 2.6 仕様書（1 本 MD）のユーザー確認タイミング

- **US-2**: 生成された仕様書をユーザーが「確認できる（表示またはダウンロード）」とある。
- **09**: 画面遷移は「トップ → 入力フォーム → サイト作成 → ローディング → 完了 or エラー」であり、**仕様書を表示する画面・ステップが明示されていない**。

**起こりうる問題**: 受け入れ条件は満たしたいが、どこで見せるか決まっておらず、実装が「入力送信前にプレビュー」「ローディング後にダウンロードリンク」などでぶれる。

**→ 対応済み**: 09 の完了ページで「仕様書（1 本マークダウン）のダウンロードリンク」を出すと定義。02 の成功時に `specMarkdown` を返すオプションを追加。

---

### 2.7 リトライ時の入力復元

- **09**: エラーページから「リトライ（入力フォームに戻って再送信）」の導線を用意する。
- **01 / US-1**: 入力内容はセッションまたは localStorage で一時保存する。
- **09**: リトライ時に**直前の入力内容をフォームに復元するか**は書かれていない。

**起こりうる問題**: リトライで空のフォームに戻ると UX が悪い。逆に「常に復元」と仕様化していないと、実装がばらつく。

**→ 対応済み**: 09 のエラーページに「リトライ時の入力復元」を追加。セッション/localStorage に保存済みの直前入力を復元して表示する旨を記載。

---

## 3. 技術選定の未確定

### 3.1 apps/web の技術

- **02-system-architecture**: 「Next.js（App Router）または React + Vite」「Tailwind 想定」とあり、**どれを採用するか**が未記載。

**起こりうる問題**: 実装者が異なる選択をすると、ディレクトリ構成・デプロイ方法・API ルートの有無が分かれる。

**→ 対応済み**: [02-system-architecture.md](./architecture/02-system-architecture.md) で「フロント（Web UI）は Next.js App Router ＋ Tailwind CSS とする」と明記。

---

### 3.2 Vercel デプロイ経路（Git 連携 vs ZIP）

- **02**: Vercel が Git 連携の場合はリモート作成・push、ZIP の場合はローカル Git のみとある。
- **03**: Vercel REST API（ZIP）または Vercel CLI。
- **05**: GitHub にリポジトリ作成・push。

**起こりうる問題**: 「GitHub に push したうえで Vercel は ZIP でデプロイする」「Vercel は GitHub リポジトリと連携してデプロイする」のどちらを初版で使うかがはっきりしておらず、deploy パッケージの責務と 03/05 の組み合わせが曖昧になる。

**→ 対応済み**: [03-vercel-deploy.md](./specs/03-vercel-deploy.md) で初版は Vercel CLI で生成ディレクトリから直接デプロイ、GitHub（05）はオプションと明記。

## 4. 成功基準・制約との整合

### 4.1 「10 分以内」の扱い

- **00-overview**: 成功基準の 1 つに「ユーザーが入力から 10 分以内にデプロイ済み URL を取得できる」とある。
- 仕様には **10 分を超えた場合の扱い**（タイムアウトでエラーにするか、ローディングを続けるか）の記載がない。

**起こりうる問題**: 実装でタイムアウトを設けると 10 分超過でエラーになり、成功基準と矛盾しない。設けないと「10 分以内」が保証されず、ユーザーがいつまで待てばよいか分からない。

**→ 対応済み**: 02 でパイプライン全体の最大待機時間を 10 分とし、超過時はエラー返却と記載。

---

### 4.2 再現性

- **00-overview**: 「同一入力に対して再現可能な生成結果が得られる（シード等で制御可能とする方針を検討）」。
- **02**: 実装は LLM の非決定的な振る舞いの影響を受けるため完全な再現は保証しない、とある。

**起こりうる問題**: 「再現可能」の度合いが仕様で定まっておらず、実装者がシードや temperature をどこまで固定するか判断に迷う。

**→ 対応済み**: 02 で「実装は LLM の非決定的な振る舞いの影響を受けるため、完全な再現は保証しない。必要に応じてシードや温度パラメータで制御する」と既にある。成功基準の「再現可能」は主に 1 本マークダウンを指し、実装コードの再現は努力目標として扱う。

---

## 5. 08 LLM 選定の未記載

- **08-llm-selection.md**: 選定結果が「（未定）」のままの場合がある。ユーザーが「LLM は決まった」と述べていても、**ドキュメントに選定結果が反映されていない**と実装がブロックする。

**起こりうる問題**: プロバイダ・モデル・API 形態が 08 に書かれていないと、core の実装（どの SDK を使うか、ストリーミングか一括か）が決められない。

**→ 対応済み**: 08 に Google / Gemini 3 Flash Preview（`gemini-3-flash-preview`）を選定結果として記載した。
