# システムアーキテクチャ

デモサイト自動生成から Vercel デプロイ・URL 共有までのデータフローとコンポーネント構成を定義する。**仮ページ駆動**：ユーザー入力 → **ロジックで**仮ページ（仕様書）生成 → **LLM（1 回のみ）**で実装 → Git リポジトリ作成 → Vercel デプロイ。

## 全体フロー

```
[ユーザー] → [Web UI] → [ロジック: 仮ページ生成] → [Markdown 仮ページ] → [LLM: 実装] → [Next.js 成果物]
     ↑            ↑              ↑                        ↑                  ↑
  入力(1-5頁等)  トリガー     packages/core           各ページの内容が     packages/core
                              (テンプレート＋組み立て)   そのまま並んだMD    (仮ページを入力に実装)

[Next.js 成果物] → [Git リポジトリ作成] → [Vercel デプロイ] → [URL 返却]
                         ↑                      ↑
                   packages/deploy 等      packages/deploy
```

1. **入力** … ユーザーが Web UI で「ページ数（1〜5）」「各ページのセクション見出し・セクション数・セクション内容・画像（添付 or テキスト or おまかせ）」「必須文言」「デザイン希望（おまかせあり）」を入力する。
2. **1 本のマークダウンの生成** … **ロジック**（テンプレートと入力スキーマの組み立て）で、**ページ構成の仕様** と **仮ページ** を **1 本のマークダウンファイル** にまとめる。LLM は使わない。内容・画像のおまかせは事前に決めた固定テキストで埋める。
3. **実装** … **LLM が 1 回だけ**その 1 本のマークダウンを読み込み、[生成サイトの技術仕様](../specs/07-generated-site-tech-spec.md)（Next.js / Tailwind / Vercel 固定）に従ってソースコードを生成する。
4. **Git リポジトリ作成** … 生成された Next.js プロジェクトで Git リポジトリを初期化（必要に応じてリモート作成）する。
5. **デプロイ** … `packages/deploy` が Git または ZIP 経由で Vercel にデプロイし、URL を取得する。
6. **URL 返却** … デプロイ完了後、生成 URL を Web UI に表示する。

## コンポーネント概要

| コンポーネント | 所在 | 役割 |
|----------------|------|------|
| Web UI | `apps/web` | 入力フォーム（ページ数・各ページのセクション見出し/数/内容/画像・必須文言・デザイン希望）、生成トリガー、仕様書表示、進行状況・エラー・URL 表示。画面構成・世界観は [09 ジェネレーター UI](../specs/09-generator-ui-spec.md) を参照。 |
| 仮ページ生成・実装 | `packages/core` または API | ユーザー入力 → **ロジック**で仮ページ生成／仮ページ → **LLM（1 回）**で Next.js 実装 |
| Git 操作 | `packages/deploy` または専用 | 生成成果物の Git 初期化・コミット（および必要に応じてリモート作成） |
| デプロイ | `packages/deploy` | Vercel API/CLI によるデプロイ、URL 取得 |
| 共有型・ユーティリティ | `packages/shared` | 入力スキーマ型、仕様書フォーマット定義、設定、ログ等 |

## データの流れ

1. **入力スキーマ**  
   - ページ数（1〜5）、サイトの説明、各ページのセクション内容、必ず入れてほしい文言。  
   - 型定義は `packages/shared` に置き、`docs/specs/` で仕様を固定する。

2. **1 本のマークダウン（仕様＋仮ページ）**  
   - ユーザー入力に基づき **ロジック** が生成する中間成果物。**ページ構成の仕様** と **仮ページ** を 1 ファイルにまとめる。画像は ref または決め文言で記載。同一入力なら同一出力。  
   - フォーマットは `docs/specs/04-markdown-spec-format.md` で定義する。

3. **実装パイプライン**  
   - LLM がその 1 本のマークダウンを入力として受け取り、[07 生成サイト技術仕様](../specs/07-generated-site-tech-spec.md)（Next.js / Tailwind / Vercel 固定）に従いソースコードを生成する。

4. **Git とデプロイ**  
   - 生成ディレクトリを Git リポジトリとして初期化し、Vercel にデプロイ（Git 連携または ZIP アップロード）。  
   - `packages/deploy` がデプロイと URL 取得を担当する。

5. **URL 表示**  
   - Web UI がデプロイ結果（成功時は URL、失敗時はエラー）を表示する。

## 技術スタック（方針）

- **フロント（Web UI）**: **Next.js（App Router）＋ Tailwind CSS** とする。生成サイト（07）とスタックを揃え、デプロイ・開発体験を統一する。
- **生成**: Node.js。仮ページは **ロジック**（テンプレート＋入力の組み立て）で生成。実装は **LLM API を 1 回だけ**用い、仮ページを入力に Next.js をコード生成。必要に応じて fs や専用ライブラリでファイル出力。
- **デプロイ**: Vercel REST API または `vercel` CLI をプログラムから呼び出す。
- **モノレポ**: pnpm workspace（または npm/yarn workspaces）で `apps/*` と `packages/*` を管理。

## セキュリティ・運用の考慮

- Vercel のトークン・プロジェクト ID は環境変数で管理し、リポジトリに含めない。
- 生成ジョブが長時間になる場合は、`apps/api` で非同期ジョブ＋ポーリングまたは WebSocket を検討する。
- レート制限・同時実行数は Vercel の制限に合わせて設計する（詳細は `specs/` で定義）。
