# システムアーキテクチャ

デモサイト自動生成から Vercel デプロイ・URL 共有までのデータフローとコンポーネント構成を定義する。**仮ページ駆動**：ユーザー入力 → マークダウン形式の仮ページ生成（LLM）→ 仮ページを読んだ LLM による実装 → Git リポジトリ作成 → Vercel デプロイ。

## 全体フロー

```
[ユーザー] → [Web UI] → [LLM: 仮ページ生成] → [Markdown 仮ページ] → [LLM: 実装] → [Next.js 成果物]
     ↑            ↑              ↑                  ↑                  ↑
  入力(1-5頁等)  トリガー     packages/core     各ページの内容が     packages/core
                                 or API           そのまま並んだMD    (仮ページを入力に実装)

[Next.js 成果物] → [Git リポジトリ作成] → [Vercel デプロイ] → [URL 返却]
                         ↑                      ↑
                   packages/deploy 等      packages/deploy
```

1. **入力** … ユーザーが Web UI で「ページ数（1〜5）」「何のサイトか」「各ページのセクション内容」「必ず入れてほしい文言」を入力する。
2. **マークダウン形式の仮ページの生成** … LLM がユーザー入力を元に、各ページの見出し・本文がそのまま並んだマークダウン（仮ページ）を生成する（役割は `packages/core` または API で実装）。
3. **実装** … LLM がその仮ページを読み込み、同じ内容・構成をイメージに合った最新技術の UI で Next.js プロジェクトとして実装する。
4. **Git リポジトリ作成** … 生成された Next.js プロジェクトで Git リポジトリを初期化（必要に応じてリモート作成）する。
5. **デプロイ** … `packages/deploy` が Git または ZIP 経由で Vercel にデプロイし、URL を取得する。
6. **URL 返却** … デプロイ完了後、生成 URL を Web UI に表示する。

## コンポーネント概要

| コンポーネント | 所在 | 役割 |
|----------------|------|------|
| Web UI | `apps/web` | 入力フォーム（ページ数・サイト説明・セクション・必須文言）、生成トリガー、仮ページ表示、進行状況・エラー・URL 表示 |
| 仮ページ生成・実装 | `packages/core` または API | ユーザー入力 → マークダウン仮ページの LLM 生成／仮ページ → Next.js 実装の LLM 呼び出し |
| Git 操作 | `packages/deploy` または専用 | 生成成果物の Git 初期化・コミット（および必要に応じてリモート作成） |
| デプロイ | `packages/deploy` | Vercel API/CLI によるデプロイ、URL 取得 |
| 共有型・ユーティリティ | `packages/shared` | 入力スキーマ型、仕様書フォーマット定義、設定、ログ等 |

## データの流れ

1. **入力スキーマ**  
   - ページ数（1〜5）、サイトの説明、各ページのセクション内容、必ず入れてほしい文言。  
   - 型定義は `packages/shared` に置き、`docs/specs/` で仕様を固定する。

2. **マークダウン形式の仮ページ**  
   - ユーザー入力に基づき LLM が生成する中間成果物。各ページの内容（見出し・本文・必須文言）がそのままマークダウンで並んだ「仮ページ」のイメージ。  
   - フォーマットは `docs/specs/04-markdown-spec-format.md` で定義する。

3. **実装パイプライン**  
   - LLM が仮ページを入力として受け取り、同じ内容・構成を Next.js プロジェクト（App Router、コンポーネント、スタイル）で実装する。  
   - テンプレートの穴埋めではなく、仮ページの見た目・内容に沿った実装を LLM が行う。

4. **Git とデプロイ**  
   - 生成ディレクトリを Git リポジトリとして初期化し、Vercel にデプロイ（Git 連携または ZIP アップロード）。  
   - `packages/deploy` がデプロイと URL 取得を担当する。

5. **URL 表示**  
   - Web UI がデプロイ結果（成功時は URL、失敗時はエラー）を表示する。

## 技術スタック（方針）

- **フロント（Web UI）**: Next.js（App Router）または React + Vite。スタイルは Tailwind CSS を想定。
- **生成**: Node.js。LLM API を用いたマークダウン仕様書の生成と、仕様書を入力とした Next.js 実装（LLM によるコード生成）。必要に応じて fs や専用ライブラリでファイル出力。
- **デプロイ**: Vercel REST API または `vercel` CLI をプログラムから呼び出す。
- **モノレポ**: pnpm workspace（または npm/yarn workspaces）で `apps/*` と `packages/*` を管理。

## セキュリティ・運用の考慮

- Vercel のトークン・プロジェクト ID は環境変数で管理し、リポジトリに含めない。
- 生成ジョブが長時間になる場合は、`apps/api` で非同期ジョブ＋ポーリングまたは WebSocket を検討する。
- レート制限・同時実行数は Vercel の制限に合わせて設計する（詳細は `specs/` で定義）。
