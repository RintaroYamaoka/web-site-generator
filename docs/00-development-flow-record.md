# 開発フロー記録（本プロジェクト）

本プロジェクトで実際にたどった**設計〜実装直前**の流れを記録する。次回や類似プロジェクトでドキュメント駆動開発を行うときの参考にする。

---

## フェーズ概要

| フェーズ | 内容 | 成果物・判断 |
|----------|------|----------------------|
| 1. 要件・方針 | 何を作るか、含めないもの | requirements/00, 01、ADR（DB なし、中身優先など） |
| 2. アーキテクチャ | 構成・データフロー・ディレクトリ | architecture/01, 02 |
| 3. 詳細仕様 | 入力・パイプライン・MD・デプロイ・UI | specs/01〜09 |
| 4. 実装前リスク調査 | 齟齬・未定義の洗い出し | implementation-risks.md |
| 5. 仕様の最適化 | リスク対応・確認事項の解消 | 各 spec の追記・修正 |
| 6. LLM 選定 | プロバイダ・モデル確定 | 08 選定結果 |
| 7. 実装フェーズ移行 | コードを書く | apps/, packages/ |

---

## 実際にやったこと（時系列イメージ）

1. **要件・目標の固定**  
   プロジェクト概要（00）、ユーザーストーリー（01）、スコープ（DB 使わない、中身を先にデザインは後）を `requirements/` と ADR に記載。

2. **アーキテクチャの定義**  
   ディレクトリ構造（01）、システム全体フロー（02）：入力 → ロジックで 1 本 MD → LLM 1 回で実装 → Git → Vercel → URL 返却。

3. **仕様の具体化**  
   - 入力スキーマ（01）、生成パイプライン（02）、Vercel（03）、1 本 MD フォーマット（04）、GitHub（05）、LLM 最適化（06）、生成サイト技術（07）、LLM 選定（08）、ジェネレーター UI（09）を順次作成・更新。
   - ジェネレーター UI は「ニアオートマタ風・無機質」「トップ→入力→ローディング→完了/エラー」「動的フォーム案」を 09 に記載。

4. **実装前リスク調査**  
   `implementation-risks.md` で、ドキュメント間の矛盾・未定義・技術選定の抜けを列挙。ユーザーに確認が必要な点（08 の記入）と、仕様で決められる点を分けた。

5. **確認が必要な点はユーザーへ、それ以外は仕様で最適化**  
   - ユーザー確認: 08 のプロバイダ・モデル（後で「無料で一番いいやつ」で Gemini 3 Flash Preview に決定）。
   - 仕様の最適化: パイプライン実行主体・タイムアウト・エラー形式、固定文言の正式定義、next-demo の役割明確化、apps/web 技術（Next.js）、初版 Vercel 経路、仕様書確認・リトライ復元などを各 spec に反映。

6. **LLM 選定の確定**  
   無料枠でコード生成向きのモデルとして **Google / Gemini 3 Flash Preview** を 08 に記載。implementation-risks の「要確認」を解消。

7. **実装フェーズ移行の判断**  
   仕様で決めるべき項目は一通り揃ったため、実装に進める状態になった。

---

## 実装フェーズに移れるか（チェックリスト）

以下の状態になっていれば **実装フェーズに移ってよい**。

- [x] 要件・ユーザーストーリーが `requirements/` にある
- [x] アーキテクチャ・ディレクトリ構造が `architecture/` で定義されている
- [x] 入力・パイプライン・MD・デプロイ・Git・LLM・生成サイト・UI の仕様が `specs/` にある
- [x] 実装前リスクが調査され、解消できるものは仕様に反映済み（implementation-risks 参照）
- [x] LLM 選定結果が 08 に記載されている（Google / Gemini 3 Flash Preview）
- [ ] （任意）画像添付時のデータフローは、実装時に 08 の API に合わせて 01 または 02 に追記する

**結論: はい、実装フェーズに移れます。** 上記チェックリストは現時点で満たしており、`docs/` を正として `apps/` と `packages/` の実装を開始してよい。

## 今後の運用

- **実装中**: 仕様とコードが食い違ったら、まず `docs/` を更新してからコードを合わせる。
- **仕様変更**: 要件・スコープの変更は `requirements/` と ADR、影響する `specs/` をまとめて更新する。
- **今回のフロー**: 設計フェーズでは「リスク調査 → 確認が必要な点はユーザー、それ以外は仕様で最適化」の一巡を入れると、実装に入る前に齟齬を減らせる。
