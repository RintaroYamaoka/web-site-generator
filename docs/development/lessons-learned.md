# 学び・振り返り（Lessons Learned）

本プロジェクトの開発で得た学びを記録・共有する。今後の実装やレビュー時の参照用。

---

## 実装の進め方：一気にやらず分割してテストしながら

### 事象

ドキュメント駆動で `docs/` を整備したところまでは良かったが、**実装を一気に行った**ことで、開発者の確認・レビュー・理解が追い付かなくなった。

### 学び

- **ドキュメントを整備したあとは、実装を小さな単位に分割し、区切りごとにテストや確認を挟む**方がよい。
- 一気に実装すると、変更量が大きくなりレビュー負荷が増し、仕様と実装の対応関係の把握も難しくなる。
- 分割して「仕様の一塊 → 実装 → 動作確認・テスト」を繰り返すことで、理解が追い付きやすく、ずれも早期に検知できる。

### 推奨する進め方

| ステップ | 内容 |
|----------|------|
| 1. 仕様の範囲を切る | 該当 spec の「今回やる部分」を明確にする（機能単位・画面単位など）。 |
| 2. 実装 | その範囲だけコードを書く。 |
| 3. 確認・テスト | 動作確認や単体テストで意図どおりか確かめる。 |
| 4. レビュー（任意） | 変更が小さいうちにレビューすると理解しやすい。 |
| 5. 次へ | 次の範囲に進む。 |

ドキュメント駆動と組み合わせる場合の流れは、**[workflow.md](./workflow.md)** の「変更は doc ファースト」を守りつつ、上記の**分割実装・テスト**を心がける。
